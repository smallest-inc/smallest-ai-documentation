This guide walks through a repeatable Lightning v3.1 evaluation: pick a dataset that matches your use case, synthesize audio, score quality with the same metrics reported on the [Performance page](/waves/documentation/text-to-speech-lightning/benchmarks/performance), and measure TTFB latency. The synthesis snippet below was **live-validated against the unified `wss://api.smallest.ai/waves/v1/tts/live` endpoint on 2026-05-21**; reference numbers from that run are in the callout below.

<Note>
**Validation results (Lightning v3.1 Standard, `robert`, 80-char English sentence):**

| Stage | Result |
|---|---|
| **TTFB** (cold conn, single request) | Standard `robert` **239 ms** • Pro `meher` **208 ms** • Pro `maverick` **228 ms** • Pro `cressida` **213 ms** |
| **Streaming RTF** | ~1.0× — server paces chunks at playback rate ([why](/waves/documentation/text-to-speech-lightning/benchmarks/metrics-overview#real-time-factor-rtf)) |
| **Synth → WAV** | 4.6 s of 44.1 kHz mono audio written to disk |
| **Whisper-small decode** | Perfect match to the input text after normalization |
| **WER / CER** | **0.0% / 0.0%** |

The whole loop — synth on WebSocket → audio chunks decoded → WAV written → Whisper transcribes → `jiwer` scores — runs end-to-end with the snippet below.
</Note>

## 1. Assemble your test set

Pick 50–200 input strings representative of your production traffic. A balanced set looks like:

- 20% short conversational ("Yeah I think so.", "Let me check.") — exercises naturalness and intonation
- 20% medium informational ("Your order has been shipped and will arrive on the 15th.") — exercises pace and pause placement
- 20% long-form narrative (3–5 sentences) — exercises prosody and breathing
- 20% domain-specific (your product names, jargon, technical terms) — exercises pronunciation
- 20% expressive (questions, exclamations, multi-clause) — exercises emotion and paralinguistics

For multilingual evaluation, repeat the structure in each language and tag entries with `language` so you can pivot results later.

```python
dataset = [
    {"text": "Yeah I think so.",                              "language": "en", "category": "conversational"},
    {"text": "Your order will arrive on the 15th.",           "language": "en", "category": "informational"},
    {"text": "क्या आप मेरा ऑर्डर ट्रैक कर सकते हैं?",                  "language": "hi", "category": "conversational"},
    {"text": "Activate the NVIDIA L40S GPU pipeline.",        "language": "en", "category": "domain"},
    # ... 50–200 entries
]
```

## 3. Synthesize + measure TTFB

This block synthesizes each entry over the streaming WebSocket and records TTFB. The pattern uses `websocket-client`'s blocking `create_connection` (validated live; simpler than the `WebSocketApp` callback API for single-request benchmarking).

```python
import os
import time
import base64
import json
import wave
from pathlib import Path
from websocket import create_connection

OUTPUT_DIR = Path("synth_outputs")
OUTPUT_DIR.mkdir(exist_ok=True)

API_KEY = os.environ["SMALLEST_API_KEY"]
WS_URL = "wss://api.smallest.ai/waves/v1/tts/live"  # unified WS — pick Pro by setting `model` body field

def synthesize(text: str, voice_id: str = "robert", model: str = "lightning_v3.1") -> dict:
    """Returns {audio_path, ttfb_ms, total_ms, audio_seconds, streaming_rtf} for one synthesis."""
    ws = create_connection(WS_URL, header=[f"Authorization: Bearer {API_KEY}"])
    req = {
        "text": text,
        "voice_id": voice_id,
        "model": model,
        "sample_rate": 44100,
        "language": "auto",
    }
    t_send = time.perf_counter()
    ws.send(json.dumps(req))

    chunks: list[str] = []
    ttfb_ms: float | None = None
    while True:
        msg = ws.recv()
        if ttfb_ms is None:
            ttfb_ms = (time.perf_counter() - t_send) * 1000
        data = json.loads(msg)
        if data.get("status") == "complete":
            break
        if "data" in data and "audio" in data["data"]:
            chunks.append(data["data"]["audio"])
    total_ms = (time.perf_counter() - t_send) * 1000
    ws.close()

    # Decode + write WAV
    pcm = b"".join(base64.b64decode(c) for c in chunks)
    audio_path = OUTPUT_DIR / f"sample_{abs(hash(text))}.wav"
    with wave.open(str(audio_path), "wb") as wf:
        wf.setnchannels(1); wf.setsampwidth(2); wf.setframerate(44100)
        wf.writeframes(pcm)
    audio_seconds = len(pcm) / (44100 * 2)

    return {
        "audio_path": audio_path,
        "ttfb_ms": ttfb_ms,
        "total_ms": total_ms,
        "audio_seconds": audio_seconds,
        "streaming_rtf": audio_seconds / (total_ms / 1000),
    }
```

To benchmark Lightning v3.1 Pro, pass `model="lightning_v3.1_pro"` and a Pro voice ID (e.g. `meher`, `maverick`, `cressida`). The same code path serves both models — the only difference is the body's `model` field.

**Wire-format note:** every server message is JSON. Audio chunks arrive as `{"data": {"audio": "<base64>"}}`. The stream terminates with `{"status": "complete"}` — no trailing audio chunk after that envelope. This is the streaming WS shape (distinct from the SSE shape on `POST /waves/v1/tts/live`).

## 4. Decode synthesized audio + score accuracy

```python
import whisper
from jiwer import wer, cer
from whisper_normalizer.english import EnglishTextNormalizer

asr = whisper.load_model("large-v3")
normalizer = EnglishTextNormalizer()

def score_sample(sample: dict, synth_result: dict) -> dict:
    decoded = asr.transcribe(str(synth_result["audio_path"]))["text"]
    ref = normalizer(sample["text"])
    hyp = normalizer(decoded)
    return {
        "text": sample["text"],
        "language": sample["language"],
        "category": sample["category"],
        "ttfb_ms": synth_result["ttfb_ms"],
        "wer": wer(ref, hyp),
        "cer": cer(ref, hyp),
        "ref": ref,
        "hyp": hyp,
    }
```

For non-English datasets, swap the normalizer for the appropriate language:

```python
from whisper_normalizer.basic import BasicTextNormalizer
normalizer = BasicTextNormalizer()  # locale-agnostic; works for Hindi, Spanish, French, etc.
```

## 5. Aggregate + report

```python
import pandas as pd

results = []
for sample in dataset:
    synth = synthesize(sample["text"], voice_id="robert")
    results.append(score_sample(sample, synth))

df = pd.DataFrame(results)
print("WER by category:")
print(df.groupby("category")["wer"].agg(["mean", "median"]))
print()
print("TTFB by language (ms):")
print(df.groupby("language")["ttfb_ms"].agg(["mean", "median", lambda x: x.quantile(0.95)]))
print()
print(f"Overall mean WER: {df['wer'].mean():.4f}")
print(f"Overall mean CER: {df['cer'].mean():.4f}")
print(f"Overall mean TTFB: {df['ttfb_ms'].mean():.1f} ms")
```

## 7. Subjective quality (Naturalness, Expressiveness, Delivery)

The Naturalness, Expressiveness, and Delivery categories on the [Performance page](/waves/documentation/text-to-speech-lightning/benchmarks/performance) are **listener-judged**, not computed. To reproduce them on your set:

- Synthesize the same input with each model you want to compare (Lightning v3.1, Lightning v3.1 Pro, competitor X).
- Run a blinded A/B/C listening test with at least 5 listeners per sample.
- Use the EmergentTTS rubric (1–5 scale per metric) or a comparable in-house rubric.
- LLM-as-a-Judge with a frontier model is a usable proxy if human listeners aren't available, but report it as automated-judged rather than human-listener-judged.

## 8. Document findings

Keep results reproducible by recording:

- Dataset SHA, audio file count, language mix
- TTS model + voice IDs used
- Whisper model version used for decoding
- Date of run + smallestai SDK version
- Aggregate WER/CER/TTFB tables broken down by category and language
- Notable failure modes (e.g., "WER spike on samples containing 'NVIDIA L40S' — fixed by pronunciation dict")
