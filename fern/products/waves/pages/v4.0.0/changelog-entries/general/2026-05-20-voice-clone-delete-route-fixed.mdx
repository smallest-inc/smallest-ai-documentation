Voice clone deletion now goes through the correct authenticated route. The fix replaces a stale path that was failing for session-authenticated callers.

**What changed under the hood:**

- Frontend delete handler now POSTs `/waves/v1/voice-cloning/delete` (JWT-auth) instead of `DELETE /waves/v1/lightning-large` (API-key-gated, incompatible with session cookies).
- More thorough S3 cleanup of the underlying clone assets.
- Publishes a `voice-updated` event to evict cached clones in Lightning replicas — preventing brief windows where a deleted voice would still synthesise.

**Migration:** no action. If you call delete from your own backend, the route is unchanged from the public API spec.
