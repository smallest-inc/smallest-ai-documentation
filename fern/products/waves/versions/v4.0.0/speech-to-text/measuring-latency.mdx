Pulse streams transcripts back while you're still sending audio. Two numbers matter for most workloads: how far behind the audio the transcript is at any moment (**transcript latency**), and how long after a speaker stops talking the final transcript arrives (**end-of-turn latency**). Voice agents are judged on the second; live-captioning UIs on the first. Averages hide tail behaviour — track p50 and p95 for whichever metric you care about.

This page defines each metric in terms of two cursors, attributes the total latency to the four components of the streaming pipeline, and gives a copy-pasteable formula + Python snippet for every measurement.

## How streaming latency is measured

Streaming latency lives in the gap between two cursors:

- **Audio cursor** — the cumulative real-time duration of audio your client has sent. You maintain this client-side by accumulating each chunk's duration as it goes out.
- **Transcript cursor** — how many seconds of audio Pulse has acknowledged transcribing. Pulse exposes this as the `end` timestamp on the last word of the most recent interim transcript, available when `word_timestamps=true` is set on the connection.

The gap between them is **transcript latency**:

```
transcript_latency = audio_cursor - transcript_cursor
```

Every metric on this page is a different sampling rule over those two cursors plus a few wire signals from Pulse (interim vs final, finalize messages, silence timeouts). Once you've internalised the cursor model, the rest of the page is just naming.

<Note>
Set `word_timestamps=true` on every connection you want to measure. Without it, Pulse doesn't tell you which audio position each transcript corresponds to, and `transcript_cursor` is permanently zero — measurements silently collapse to nonsense.
</Note>

## Which metric matches your workload

There are three useful streaming metrics and one pre-recorded metric. Pick the one that answers the product question you're trying to answer.

| Metric | Answers | Use case |
|---|---|---|
| Transcript latency | How far behind the audio cursor is the transcript cursor right now? | Live captioning, real-time analytics, conversation displays |
| End-of-turn (EOT) latency | How long after a speaker stops talking does the final transcript arrive? | Voice agents — drives perceived responsiveness |
| Time to first partial | How long after audio starts does the first interim transcript arrive? | Session-startup health, first-paint UX |
| RTFx (pre-recorded) | What fraction of real-time did Pulse process this file at? | Batch transcription, cross-vendor accuracy benchmarks |

EOT is sometimes called "end of utterance" — same metric, different vocabulary. Voice-agent builders typically search for EOT, so we lead with that name on this page.

## Latency components

When a latency number is higher than expected, the cause lives in one of four places. Measure each in isolation to attribute.

```
client                  network                 server                  network                 client
  │                        │                       │                       │                        │
  ├── encode chunk ───────►│                       │                       │                        │
  │                        ├── transit ──────────► │                       │                        │
  │                        │                       ├── model inference ──► │                        │
  │                        │                       │                       ├── transit ───────────► │
  │                        │                       │                       │              decode + render
  └─── client overhead ────┴─── network ───────────┴─── transcription ─────┴─── network ────────────┘
       (encode/send loop)       (each direction)        (steady state)         (each direction)
```

### Connection setup

A one-time cost paid on the first WebSocket frame of every new session: TLS handshake, HTTP upgrade, and server worker selection. Subsequent messages on the same socket don't pay this.

Approximate with `curl`:

```bash
curl -sSf -w "tls=%{time_appconnect}s upgrade=%{time_starttransfer}s\n" \
  -so /dev/null https://api.smallest.ai
```

Typical: **20–80 ms** depending on geographic distance to `api.smallest.ai` (Mumbai) or `api.us.smallest.ai` (Oregon). First request after a long idle window may also re-warm a server worker — exclude it from averages.

### Network transit

Per-message TCP transit plus any server-side queue wait. Measure the round trip by sending a chunk and timing until the first transcript arrives back, then halving:

```
network_rtt ≈ (t_first_response - t_chunk_sent)
network_one_way ≈ network_rtt / 2
```

Typical: **5–30 ms one-way** within the same region, **80–200 ms** cross-region. If your client is in `us-east-1` and you're connecting to `api.smallest.ai` (Mumbai), you're paying network — pin the region.

### Server transcription

The model inference time. Steady-state contribution to transcript latency:

```
server_transcription ≈ transcript_latency - 2 * network_one_way - client_overhead
```

Typical: **sub-100 ms** at 1 concurrency, scaling to **~300 ms** at 100 concurrent requests on the production deployment. See the [Pulse model card](/waves/model-cards/speech-to-text/pulse) for headline numbers and concurrency curves.

### Client overhead

Encoding, the send loop, the receive loop, and any UI render between `ws.recv()` and the transcript appearing on screen. Easy to overlook because it doesn't show up in the wire.

Measure as wall-clock between `chunk_ready` and `ws.send()`, plus between `ws.recv()` and `on_transcript()`. Typical: **1–10 ms** for a well-written client; can balloon to hundreds of milliseconds if you're doing CPU-heavy work on the main thread or rendering through a slow framework.

<Note>
Chunk size affects two components at once. Chunks below **10 ms** flood the server with WebSocket frames; per-frame overhead dominates. Chunks above **500 ms** turn the stream into a batch and add their own buffer latency. **20–100 ms per chunk** is the practical range for low-latency streaming.
</Note>

## Measuring each metric

### Transcript latency

Sample continuously across the session — every time an interim transcript (`is_final=false`) arrives, take the cursor difference:

```python
import json

audio_cursor = 0.0           # seconds of audio sent so far
transcript_cursor = 0.0      # seconds of audio Pulse has transcribed

def on_chunk_sent(num_bytes, sample_rate=16000, bytes_per_sample=2):
    """Call after every binary chunk you send to the server."""
    global audio_cursor
    audio_cursor += num_bytes / (sample_rate * bytes_per_sample)

def on_message(raw):
    """Call on every JSON frame Pulse sends back."""
    global transcript_cursor
    msg = json.loads(raw)
    if msg.get("type") != "transcription":
        return
    # Use ONLY interim transcripts. Finals are delayed by the server's
    # accuracy buffer and don't reflect real-time perceived latency.
    if msg.get("is_final"):
        return
    words = msg.get("words", [])
    if words:
        transcript_cursor = words[-1]["end"]
    latency_ms = (audio_cursor - transcript_cursor) * 1000
    print(f"transcript_latency = {latency_ms:.0f} ms")
```

Two rules to follow:

1. **Sample only on interim transcripts.** Pulse emits `is_final=true` after an internal accuracy buffer fills, which trades a few hundred milliseconds for a better final transcript. Treating finals as a latency signal makes the number look worse than what users perceive. See [Realtime response format](/waves/documentation/speech-to-text-pulse/realtime-web-socket/response-format) for the buffer behaviour.
2. **Set `word_timestamps=true`** on the connection. Without it the `end` field is missing and `transcript_cursor` stays at zero.

### End-of-turn (EOT) latency

EOT is the time between a speaker stopping and the corresponding finalised transcript arriving:

```
eot_latency = t_final_arrived - t_audio_silence_started
```

Pulse doesn't know when your speaker stopped — your client does. There are three patterns; the **finalize-on-VAD** pattern is the recommended one for voice agents.

**Pattern A — finalize on VAD (recommended).** Run a client-side voice activity detector. When it reports silence, send a `finalize` control message immediately and measure time until the next `is_final=true` returns.

```python
import json, time

silence_started_at = None

# When client-side VAD reports silence:
silence_started_at = time.perf_counter()
ws.send(json.dumps({"type": "finalize"}))

def on_message(raw):
    global silence_started_at
    msg = json.loads(raw)
    if msg.get("is_final") and silence_started_at is not None:
        eot_ms = (time.perf_counter() - silence_started_at) * 1000
        print(f"eot_latency = {eot_ms:.0f} ms")
        silence_started_at = None
```

**Pattern B — server-side silence timer.** Configure `eou_timeout_ms` on the connection. Pulse emits `is_final=true` automatically after that many ms of silence in the inbound audio. EOT in this mode approximates `eou_timeout_ms + transcript_latency` — easier to set up, but EOT can never beat the timer.

**Pattern C — VAD without finalize.** Mark the silence timestamp on your VAD and wait passively for the next `is_final=true`. Equivalent to Pattern B without the explicit timer — useful only if you're avoiding `finalize` for protocol reasons.

### Time to first partial

Elapsed time between sending the first chunk and the first interim transcript arriving back:

```python
import time, json

t_start = time.perf_counter()
got_first_partial = False

def on_message(raw):
    global got_first_partial
    msg = json.loads(raw)
    if msg.get("type") == "transcription" and not got_first_partial:
        ttfp_ms = (time.perf_counter() - t_start) * 1000
        print(f"time_to_first_partial = {ttfp_ms:.0f} ms")
        got_first_partial = True
```

This is a startup health check, not a steady-state metric. The first session on a fresh process pays the connection-setup tax (see [Latency components](#latency-components)). **Always exclude the first session from any average.**

## Pre-recorded latency (RTFx)

For `POST /waves/v1/stt/`, the audio arrives in one request body and the transcript in one response. The natural metric is wall-clock time from request start to transcript received, normalised by audio duration so file lengths don't dominate:

```
RTFx = audio_duration_seconds / processing_time_seconds
```

An RTFx of 50 means a 60-second file came back in 1.2 seconds. **Higher is better.** RTFx is the standard cross-vendor comparison for pre-recorded transcription because it normalises across audio length.

```python
import time, requests, os

t0 = time.perf_counter()
response = requests.post(
    "https://api.smallest.ai/waves/v1/stt/",
    params={"model": "pulse-pro", "language": "en"},
    headers={
        "Authorization": f"Bearer {os.environ['SMALLEST_API_KEY']}",
        "Content-Type": "application/octet-stream",
    },
    data=open("call.wav", "rb").read(),
    timeout=120,
)
response.raise_for_status()
wall_clock = time.perf_counter() - t0
audio_duration = response.json()["metadata"]["duration"]
print(f"RTFx (wall-clock) = {audio_duration / wall_clock:.1f}")
```

The response includes `metadata.processing_time_ms` — the server's internal measurement, which **excludes** network round-trip and request body upload. Use it for attribution: if wall-clock RTFx is far worse than server-time RTFx, the bottleneck is your network or upload bandwidth, not the model.

## Common measurement pitfalls

- **Sampling latency from `is_final=true`.** Finals are intentionally delayed by Pulse's accuracy buffer. They look slow because they are — by design. Use interims.
- **Forgetting `word_timestamps=true`.** `transcript_cursor` stays at zero. Code runs silently, numbers are meaningless.
- **Including the first session in averages.** It pays a one-time TLS + worker warm-up tax that no real user will experience after the first turn.
- **Comparing across regions without pinning.** Same audio + same model + different client region can swing transcript latency by 100+ ms. Pin the region when benchmarking the model.
- **Treating EOT and transcript latency as the same number.** They measure different things. A 50 ms transcript latency and a 400 ms EOT can coexist (and usually do — finalize messages traverse the same network round-trip the transcript does).
