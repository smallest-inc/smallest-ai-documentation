Streaming speech-to-text latency is the time between an event in the audio (a word spoken, an utterance ending) and the corresponding event in the transcript stream (the word appearing, the final transcript arriving). Because audio arrives incrementally, there isn't one number that captures the whole experience — different applications need different metrics.

For real-time applications like voice agents and live captioning, latency drives perceived responsiveness. A voice agent cares about how quickly the model returns a finalised transcript after the user stops speaking. A live captioning UI cares about how far behind the audio the transcript is at any moment. These are different measurements, and a single average can hide problems with either one.

This page defines each metric, shows how to calculate it against the wire signals Pulse emits, and walks through the common mistakes that inflate the numbers.

## Streaming metrics

There are three useful metrics for `WSS /waves/v1/stt/live` (and the legacy `/waves/v1/pulse/get_text`). Choose the one that matches the product question you're answering.

| Metric | Question it answers | Use case |
|---|---|---|
| Transcript latency | How far behind the audio cursor is the transcript cursor? | Live captioning, real-time analytics, conversation displays |
| End-of-utterance latency | How long after the user stops speaking does the final transcript arrive? | Voice agents — drives perceived responsiveness |
| Time to first partial | How long after audio starts does the first transcript text appear? | Session-startup health checks, first-paint UX |

### Transcript latency

Transcript latency tells you how far behind the audio the transcription is at any given moment. Sample it continuously across the session — every time an interim transcript (`is_final=false`) arrives, compute:

```
transcript_latency = audio_cursor_seconds - transcript_cursor_seconds
```

`audio_cursor_seconds` is the cumulative real-time duration the client has sent. Maintain it client-side by accumulating each chunk's duration as you send. `transcript_cursor_seconds` is how many seconds of audio the server has acknowledged transcribing. Pulse exposes this through the `end` timestamp on the last word of the most recent interim transcript, available when `word_timestamps=true` is on.

```python
import json

audio_cursor = 0.0           # seconds of audio sent so far
transcript_cursor = 0.0      # seconds of audio Pulse has transcribed

# After sending each binary chunk
def on_chunk_sent(num_bytes, sample_rate=16000, bytes_per_sample=2):
    global audio_cursor
    audio_cursor += num_bytes / (sample_rate * bytes_per_sample)

# On every received transcription message
def on_message(msg):
    global transcript_cursor
    j = json.loads(msg)
    if j.get("type") != "transcription":
        return
    # Use ONLY interim transcripts. Finals are delayed by the server's
    # internal accuracy buffer and don't reflect the real-time experience.
    if j.get("is_final"):
        return
    words = j.get("words", [])
    if words:
        transcript_cursor = words[-1]["end"]
    latency = audio_cursor - transcript_cursor
    print(f"transcript_latency = {latency * 1000:.0f} ms")
```

Two things to get right when computing this metric.

First, sample only on interim transcripts. Pulse emits `is_final=true` after an internal accuracy buffer fills, which trades a few hundred milliseconds for a better final transcript. Treating finals as a latency signal makes the number look worse than what users actually perceive. See the [Real-time response format](/waves/documentation/speech-to-text-pulse/realtime-web-socket/response-format) page for the buffer behaviour.

Second, set `word_timestamps=true` on the connection. Without it the server doesn't tell you which audio position each transcript chunk corresponds to, and there's no way to compute `transcript_cursor`.

### End-of-utterance latency

End-of-utterance (EOU) latency is the time between when the user stops speaking and when the corresponding finalised transcript arrives. It's the metric voice agents are judged on.

```
eou_latency = t_final_arrived - t_audio_silence_started
```

The client doesn't natively know when the user stopped speaking, so you measure it one of three ways.

Run a client-side voice activity detector. When the VAD detects silence, record the timestamp and wait for the next `is_final=true` to arrive. Subtract.

```python
import json, time, websocket

silence_started_at = None

# When client-side VAD says "user paused":
silence_started_at = time.perf_counter()
ws.send(json.dumps({"type": "finalize"}))

def on_message(raw):
    global silence_started_at
    msg = json.loads(raw)
    if msg.get("is_final") and silence_started_at is not None:
        eou_latency_ms = (time.perf_counter() - silence_started_at) * 1000
        print(f"eou_latency = {eou_latency_ms:.0f} ms")
        silence_started_at = None
```

Or send `{"type":"finalize"}` immediately after your VAD detects silence and measure time until the next `is_final=true` lands. This is the recommended pattern for voice agents because it forces a finalisation rather than waiting for Pulse's internal silence timer.

Or configure Pulse's silence timeout via `eou_timeout_ms`. The server emits `is_final=true` automatically after that many milliseconds of silence. EOU latency in this mode approximates `eou_timeout_ms + transcript_latency`.

### Time to first partial

Time to first partial is the elapsed time between the first audio chunk being sent and the first interim transcript arriving back. It's a session-startup health check, not a steady-state measure.

```python
import time, json

t_start = time.perf_counter()
got_first_partial = False

def on_message(raw):
    global got_first_partial
    msg = json.loads(raw)
    if msg.get("type") == "transcription" and not got_first_partial:
        ttfp_ms = (time.perf_counter() - t_start) * 1000
        print(f"time_to_first_partial = {ttfp_ms:.0f} ms")
        got_first_partial = True
```

The first session on a fresh process pays a one-time cost from the WebSocket TLS handshake and server worker warm-up. Subsequent sessions on the same client reuse the connection. Exclude the first session from any average.

## Pre-recorded latency

For `POST /waves/v1/stt/`, the audio arrives in a single request body and the transcript comes back in a single response. The natural metric is wall-clock time from request start to transcript received, normalised by audio duration so file sizes don't dominate.

### RTFx — Real-Time Factor

```
RTFx = audio_duration_seconds / processing_time_seconds
```

An `RTFx` of 50 means a 60-second file came back in 1.2 seconds. Higher is better. RTFx is the standard cross-vendor comparison metric for pre-recorded transcription because it normalises across audio length.

```python
import time, requests, os

t0 = time.perf_counter()
response = requests.post(
    "https://api.smallest.ai/waves/v1/stt/",
    params={"model": "pulse-pro", "language": "en"},
    headers={
        "Authorization": f"Bearer {os.environ['SMALLEST_API_KEY']}",
        "Content-Type": "application/octet-stream",
    },
    data=open("call.wav", "rb").read(),
    timeout=120,
)
response.raise_for_status()
elapsed = time.perf_counter() - t0
audio_duration = response.json()["metadata"]["duration"]
print(f"RTFx = {audio_duration / elapsed:.1f}")
```

The response includes a `metadata.processing_time_ms` field. That's the server's internal measurement — it excludes network round-trip and request body upload. Use it to attribute slowdowns: if wall-clock RTFx is far worse than server `processing_time_ms` suggests, the bottleneck is network or upload bandwidth, not the model.

## Component breakdown

When a streaming latency number is higher than expected, the cause lives in one of four places. Measure each in isolation to attribute.

| Component | What it is | How to measure |
|---|---|---|
| Network connection | One-time TLS + WebSocket upgrade | Approximate with `curl -sSf -w "tls: %{time_appconnect} total: %{time_starttransfer}\n" -so /dev/null https://api.smallest.ai` |
| Network per-message | TCP transit + server queue wait | Round-trip ≈ time from `ws.send()` to first echo back, divided by 2 |
| Transcription | Server-side model inference | Total `transcript_latency` minus network transit |
| Client | Encoding, send loop, recv loop, UI render | Wall-clock between `chunk_ready` and `ws.send()`, plus between `ws.recv()` and `on_transcript()` |

Audio chunking matters too. Chunks below 10 ms flood the server with WebSocket frames and per-message overhead dominates. Chunks above 500 ms turn the stream into a batch and add their own buffer latency. 20 to 100 ms per chunk is the practical range for low-latency streaming.

## Common measurement pitfalls

Sampling latency from `is_final=true` transcripts. Finals are delayed by Pulse's accuracy buffer. They look slow if you measure latency from them. Use interim (`is_final=false`) transcripts instead.

Forgetting to set `word_timestamps=true`. Without word timestamps, `transcript_cursor` is always zero and `transcript_latency` collapses to `audio_cursor`. The measurement code silently produces meaningless numbers.

Including the first session in averages. The first session on a process pays a one-time TLS + worker warm-up tax. Average across the second-and-later sessions.

Comparing across networks. The same audio file streamed from a different region will pick up extra transit time. Pin the test region when comparing model performance against transport.

Long idle gaps. If a session sits idle long enough, the server worker may spin down. The next request after that pays a re-warm cost. Don't include it in averages.

## What to measure for which use case

| Building... | Primary metric | Secondary |
|---|---|---|
| Live captioning UI | Transcript latency (p50, p95) | Time to first partial |
| Voice agent | End-of-utterance latency (p50, p95) | Transcript latency |
| Post-call analytics on uploaded files | RTFx | Wall-clock end-to-end |
| Real-time analytics dashboard | Transcript latency | EOU at conversation segment boundaries |
