This is the canonical reference for the Hydra WebSocket. For a 50-line getting-started, see the [Quickstart](/waves/documentation/speech-to-speech-hydra/quickstart).

## Connection

| | |
|---|---|
| Endpoint | `wss://api.smallest.ai/waves/v1/s2s` |
| Query parameters | `model=hydra` (required), `api_key=<SMALLEST_API_KEY>` (required) |
| Frame format | JSON (UTF-8 text frames). Binary frames are not used. |
| Input audio | Base64-encoded PCM16, signed little-endian, mono, **16 kHz** |
| Output audio | Base64-encoded PCM16, signed little-endian, mono. Sample rate announced in `session.configured` (**currently 48000 Hz**). |
| Authentication | `api_key` query parameter on the connect URL |
| Idle timeout | ~30 seconds of no traffic from either side. Reconnect to resume. |

### Connection rejection and close codes

- **Bad `api_key`** is rejected during the WebSocket *handshake* with **HTTP 401** — the upgrade never completes, so there is no `error` event and no close code on the wire. Surface the credential failure to the user before retrying.
- **Server at capacity** returns close code **`1013`**, preceded by an `error` event with `code: "server_full"`. Back off with jitter and retry.
- **Normal close** is code **`1000`**, including the ~30 s server-side idle timeout. Reconnect if the user is still active.

## Session lifecycle

A single WebSocket connection corresponds to one voice session.

```
client                                            server
  │  ── WebSocket connect ──▶                       │
  │  ◀─── session.created ─────────────────────────  │
  │  ──── session.configure ─────────────────────▶   │   one-shot
  │  ◀─── session.configured ──────────────────────  │   echoes effective config
  │  ──── input_audio_buffer.append ─────────────▶   │   stream mic audio
  │  ◀─── input_audio_buffer.speech_started ───────  │
  │  ◀─── conversation.item.added (user) ──────────  │
  │  ◀─── input_audio_buffer.speech_stopped ───────  │
  │  ◀─── conversation.item.done (user) ───────────  │
  │  ◀─── response.created ────────────────────────  │
  │  ◀─── conversation.item.added (assistant) ─────  │
  │  ◀─── response.output_audio.delta ─────────────  │   base64 PCM16 chunks
  │  ◀─── response.output_audio.delta ─────────────  │
  │  ◀─── response.output_audio.done ──────────────  │
  │  ◀─── conversation.item.done (assistant) ──────  │
  │  ◀─── response.done ───────────────────────────  │
```

### Protocol invariants

- **Handshake is one-shot.** After `session.created`, the server waits for exactly one `session.configure` before accepting `input_audio_buffer.append` or any response-related event. Subsequent `session.configure` frames are ignored; use `session.update` for mid-session changes.
- **Stream audio continuously.** There is no manual `commit` or `end-of-turn`. Hydra decides when a turn ends.
- **Full-duplex.** If the user speaks while the model is responding, the in-flight response is cancelled (`response.done` with `status: "cancelled"` and `status_details.reason: "interrupted"`) and a fresh turn begins.
- **No transcript on the wire.** Audio bytes are the payload. `item.content[].text` is `null` for audio items. Transcribe client-side or via the [Pulse STT API](/waves/documentation/speech-to-text-pulse/overview) if text is required.

## Authentication

```
wss://api.smallest.ai/waves/v1/s2s?model=hydra&api_key=<SMALLEST_API_KEY>
```

Treat the key as a secret. For browser-based applications, mint a short-lived token server-side rather than shipping a long-lived key to the client.

If the key is missing or invalid, the upgrade is rejected with **HTTP 401** during the WebSocket handshake — the connection never opens, so no event frames or close codes are emitted on the wire. Detect this from your client's connection error rather than expecting an `error` event.

## Audio formats

### Input (client → server)

PCM16, signed little-endian, mono, **16 kHz**, base64-encoded inside `input_audio_buffer.append`. Recommended chunk size: **20–40 ms (640–1280 samples)**. Use an `AudioWorklet` in the browser — the deprecated `ScriptProcessorNode` is OK for a prototype but not for production.

```js
function floatTo16BitPCM(float32) {
  const out = new Int16Array(float32.length);
  for (let i = 0; i < float32.length; i++) {
    const s = Math.max(-1, Math.min(1, float32[i]));
    out[i] = s < 0 ? s * 0x8000 : s * 0x7fff;
  }
  return out;
}

function arrayBufferToBase64(buf) {
  let bin = "";
  const bytes = new Uint8Array(buf);
  for (let i = 0; i < bytes.length; i++) bin += String.fromCharCode(bytes[i]);
  return btoa(bin);
}
```

### Output (server → client)

PCM16, signed little-endian, mono. The sample rate is announced in `session.configured.session.output_audio_sample_rate` and is currently **48000 Hz** — **read it from the event rather than hardcoding**, since the server may negotiate other rates in the future. Each chunk arrives base64-encoded inside `response.output_audio.delta`.

If `response.created` arrives before the previous response's `response.output_audio.done`, drop any still-scheduled audio buffers from the previous response — the user has barged in.

## Event reference

All events are JSON objects with a `type` discriminator. Server events carry a server-minted `event_id`. Client events should include a client-generated `event_id` (any unique string, UUID recommended) so server errors can reference the originating frame.

### Server → Client

| `type` | Description | Payload |
|---|---|---|
| `session.created` | First message after the WebSocket opens. | `{ session_id }` |
| `session.configured` | Emitted once in response to `session.configure`. Echoes the effective config including the negotiated `output_audio_sample_rate`. | `{ session: SessionConfig }` |
| `session.updated` | Echo of a mid-session `session.update`. Contains only fields that were applied. | `{ session: { ... } }` |
| `input_audio_buffer.speech_started` | Hydra detected the start of user speech. `item_id` is the ID of the user message item that will be created. | `{ audio_start_ms, item_id }` |
| `input_audio_buffer.speech_stopped` | Hydra detected the end of user speech. | `{ audio_end_ms, item_id }` |
| `conversation.item.added` | A new item entered the conversation. | `{ previous_item_id?, item: ConversationItem }` |
| `conversation.item.done` | An item reached its final state. | `{ previous_item_id?, item: ConversationItem }` |
| `response.created` | The model started generating a response. | `{ response: { id } }` |
| `response.output_audio.delta` | Streamed audio chunk. Decode base64 → PCM16 and play at the negotiated rate. | `{ response_id, item_id, output_index, content_index, delta }` |
| `response.output_audio.done` | Audio streaming for the current output item is complete. | `{ response_id, item_id, output_index, content_index }` |
| `response.function_call_arguments.delta` | Streaming JSON args for a tool call. Concatenate per `call_id`. | `{ response_id, call_id, name?, delta }` |
| `response.function_call_arguments.done` | Tool call complete. The client is responsible for executing the tool. | `{ response_id, call_id, name, arguments }` |
| `response.done` | The response turn finished — successfully, cancelled, incomplete, or failed. | `{ response: { id, status, status_details?, output, usage? } }` |
| `error` | A recoverable error. The connection remains usable unless followed by a close frame. | `{ error: { type, code, message, param?, event_id? } }` |

<Note>
**Hydra does not emit transcript events.** Specifically, there is no `response.output_audio_transcript.*` and no `conversation.item.input_audio_transcription.completed`.
</Note>

### Client → Server

| `type` | When to send | Payload |
|---|---|---|
| `session.configure` | **Exactly once**, immediately after `session.created`. The server will not accept audio until this is processed. | `{ session: SessionConfig }` |
| `session.update` | Mid-session live patch. Currently only `tools` is honoured. | `{ session: { tools? } }` |
| `input_audio_buffer.append` | Continuously while the microphone is open. | `{ audio: base64-pcm16 }` |
| `conversation.item.create` | Post the result of a tool call back to the model. | `{ item: { type: "function_call_output", call_id, output } }` |
| `response.create` | After all tool outputs for the turn are posted, request narration. | `{ response?: { ... } }` (body optional) |
| `response.cancel` | Programmatically cancel an in-flight response. | `{ response_id? }` |

Unrecognised event types are ignored by the server. Don't rely on this for behaviour — it's a forward-compatibility guard.

## `SessionConfig`

Sent inside `session.configure` and echoed inside `session.configured`. All fields are optional; omitted fields fall back to the server default.

| Field | Type | Description |
|---|---|---|
| `instructions` | string | Persona / system prompt. Output-shaping rules (style tags, formatting) are managed server-side. |
| `voice` | string | Voice identifier. Currently: `leda`, `aoede`, `autonoe`, `charon`, `fenrir`, `zephyr`. |
| `tools` | array of `Tool` | Function-calling schemas the model may invoke. |
| `output_audio_format` | string | Currently only `"pcm16"`. |
| `output_audio_sample_rate` | integer | Output sample rate. Server selects a sensible default (currently `48000`); the value echoed in `session.configured` is the source of truth for playback. |
| `generate_initial_response` | boolean | When `true`, the model speaks first. Honoured only at handshake. |

### `Tool` shape

```json
{
  "type": "function",
  "name": "get_weather",
  "description": "Look up current weather for a city.",
  "parameters": {
    "type": "object",
    "properties": { "city": { "type": "string" } },
    "required": ["city"]
  }
}
```

### Mid-session updates

`session.update` is honoured during a session, but **only the `tools` field is applied today**. Persona, voice, and audio formats are frozen at handshake; changes to those require a fresh connection.

```json
{ "type": "session.update", "session": { "tools": [ /* ... */ ] } }
```

The server replies with `session.updated` containing only the fields it actually applied. A no-op patch produces no echo.

## `ConversationItem`

Most conversation events carry a `ConversationItem`. The shape is intentionally flat — every field is optional, presence is dictated by `type`.

```json
{
  "id": "item_...",
  "type": "message" | "function_call" | "function_call_output",
  "role": "user" | "assistant" | "system",
  "status": "in_progress" | "completed" | "incomplete",
  "content": [
    { "type": "input_audio" | "input_text" | "output_audio" | "output_text", "text": null }
  ],
  "call_id": "call_...",         // function_call / function_call_output only
  "name": "get_weather",          // function_call only
  "arguments": "{...json...}",    // function_call only
  "output": "12°C, light rain"   // function_call_output only
}
```

For audio message items, `content[].text` is **always `null`** — audio bytes are the payload. The `text` field exists for `function_call_output` and other text-bearing items.

Discarded user turns (e.g. brief mic noise that did not trigger a response) arrive as `conversation.item.done` with `status: "incomplete"` and should be ignored.

## `response.done`

```json
{
  "type": "response.done",
  "event_id": "sv_...",
  "response": {
    "id": "resp_...",
    "status": "completed" | "cancelled" | "incomplete" | "failed",
    "status_details": {
      "reason": "interrupted" | "client_cancelled" | "max_output_tokens" | "content_filter",
      "type": "...",
      "error": { "code": "...", "type": "..." }
    },
    "output": [ /* ConversationItem */ ],
    "usage": { "input_tokens": 0, "output_tokens": 0, "total_tokens": 0 }
  }
}
```

| `status` | Interpretation |
|---|---|
| `"completed"` | Turn finished normally. `status_details` is omitted. |
| `"cancelled"` • `reason: "interrupted"` | The user barged in. |
| `"cancelled"` • `reason: "client_cancelled"` | The client sent `response.cancel`. |
| `"incomplete"` | Stop condition reached (`max_output_tokens`, `content_filter`). |
| `"failed"` | Internal error. `status_details.error` carries `{ code, type }`. |

`usage` is populated when token counts are available. For per-turn latency, measure on the client (`response.created` arrival vs the first `response.output_audio.delta`).

## Tool calling

Hydra is a voice model — it doesn't execute tools. The client declares schemas in `session.configure`, Hydra decides when to call them and streams the arguments JSON, and the client executes the tool locally and posts the result back.

### Event flow

```
client: session.configure                       (with tools: [...])
  ...user speaks: "what's the weather in Paris?"...
server: response.created
server: conversation.item.added                 (role=assistant, in_progress)
server: response.function_call_arguments.delta  { call_id, name, delta: "{\"cit"  }
server: response.function_call_arguments.delta  { call_id, delta: "y\":\"Paris\"}" }
server: response.function_call_arguments.done   { call_id, name, arguments: "{\"city\":\"Paris\"}" }

client: conversation.item.create                { item: { type: "function_call_output",
                                                          call_id, output: "12°C, light rain" } }
client: response.create

server: response.created                        (narration response)
server: response.output_audio.delta             ...  (bot narrates the result)
server: response.output_audio.done
server: response.done
```

### Implementation

```python
# Debounce response.create so that parallel tool calls in a single turn
# don't trigger narration before every tool output has been posted.
pending_response_create: asyncio.Task | None = None
DEBOUNCE_MS = 200

async def schedule_response_create():
    """Send one response.create per turn, ~200 ms after the last tool output."""
    await asyncio.sleep(DEBOUNCE_MS / 1000)
    await ws.send(json.dumps({"type": "response.create"}))

async for raw in ws:
    evt = json.loads(raw)
    if evt["type"] == "response.function_call_arguments.done":
        args = json.loads(evt["arguments"])
        result = run_tool(evt["name"], args)
        await ws.send(json.dumps({
            "type": "conversation.item.create",
            "item": {
                "type": "function_call_output",
                "call_id": evt["call_id"],
                "output": result if isinstance(result, str) else json.dumps(result),
            },
        }))
        # (Re-)schedule a single response.create after the last tool output.
        # If more tool outputs arrive within DEBOUNCE_MS, the previous task
        # is cancelled and replaced — only the final one fires.
        if pending_response_create and not pending_response_create.done():
            pending_response_create.cancel()
        pending_response_create = asyncio.create_task(schedule_response_create())
    elif evt["type"] == "response.created":
        # New turn started — the previous turn's narration is in flight.
        pending_response_create = None
```

If the model calls multiple tools in a single turn, the server emits one `response.function_call_arguments.done` per call. Post one `conversation.item.create` per tool output, then send **a single** `response.create` to trigger narration — sending `response.create` after every tool output causes the server to start narrating before all results are in. The debounce above (`~200 ms`) batches parallel tool calls into one narration request. For single-tool turns the debounce adds 200 ms of latency, which is well below the model's own time-to-first-audio.

If the tool output is not posted within the server's timeout window, an `error` with `code: "tool_response_timeout"` is emitted and the turn is abandoned.

## Interruption and cancellation

### User interruption (automatic)

Because the channel is full-duplex, the user may speak at any time — including over the model. When this happens:

1. The server emits `input_audio_buffer.speech_started` for the new user turn.
2. The in-flight response is cancelled:

```json
{
  "type": "response.done",
  "response": {
    "id": "resp_...",
    "status": "cancelled",
    "status_details": { "reason": "interrupted" }
  }
}
```

3. The new turn proceeds normally.

On the client, **stop and disconnect any scheduled `AudioBufferSourceNode`s from the previous response**. Closing the `AudioContext` alone leaves a tail.

### Programmatic cancellation

```json
{ "type": "response.cancel" }
```

The server replies with `response.done` carrying `status: "cancelled"` and `status_details.reason: "client_cancelled"`. The frame is a no-op if no response is in flight.

## Bot speaks first

To have the model deliver an opening line before the user has said anything, set `generate_initial_response: true` on `session.configure`. The opening line is steered by `instructions`.

```json
{
  "type": "session.configure",
  "session": {
    "instructions": "You are a hotel concierge. Open the call by greeting the guest warmly and asking how you can help.",
    "voice": "leda",
    "generate_initial_response": true
  }
}
```

Immediately after `session.configured`, the standard `response.created` → audio deltas → `response.done` sequence is emitted, with no preceding `input_audio_buffer.speech_started`.

## Style tags

`<style>` tags can be used in `instructions` to request specific emotional rendering for spoken output:

```
<style emotion="excited">That's incredible news!</style>
```

Style tags are interpreted server-side and the audio you receive already reflects the requested prosody. The tags themselves do not appear in any wire payload.

## Errors

`error` events follow this shape:

```json
{
  "type": "error",
  "event_id": "sv_...",
  "error": {
    "type": "invalid_request_error",
    "code": "invalid_audio",
    "message": "audio frame too small (40 bytes, need 320)",
    "param": "audio",
    "event_id": "evt_..."
  }
}
```

When present, `error.event_id` is the `event_id` of the client frame that triggered the error, allowing direct correlation.

| `code` | Meaning |
|---|---|
| `invalid_request_error` | Schema validation failure — unknown field, missing `type`, wrong type, malformed JSON, unsupported audio format. |
| `invalid_audio` | An `input_audio_buffer.append` payload was not valid base64 or was not PCM16. Session continues. |
| `tool_response_timeout` | Tool schemas were declared, the model called a tool, and the server did not receive `function_call_output` + `response.create` within the timeout window. Turn is abandoned. |
| `server_full` | Sent immediately before a `1013` close — the server is at capacity. |
| `internal_error` | Unhandled server-side exception. Reconnect and surface to support if persistent. |

Errors are **non-fatal unless followed by a close**. Treat them as diagnostics.

<Note>
Voice IDs that aren't in the supported list are accepted by the server (no `error` event) and fall through to the default voice. Always validate voice IDs client-side against the [supported list](#sessionconfig) before sending.
</Note>

## Browser microphone example

A minimal browser client. For production (gapless playback, barge-in, AudioWorklet), see the [reference client](https://github.com/smallest-inc/hydra_agents).

```js
const apiKey = "<SMALLEST_API_KEY>";
const ws = new WebSocket(
  `wss://api.smallest.ai/waves/v1/s2s?model=hydra&api_key=${apiKey}`,
);
let micCtx, playCtx, ttsRate = 48000, playCursor = 0;

ws.onmessage = async (ev) => {
  const evt = JSON.parse(ev.data);
  switch (evt.type) {
    case "session.created":
      ws.send(JSON.stringify({
        type: "session.configure",
        session: { instructions: "You are a friendly assistant.", voice: "leda" },
      }));
      break;
    case "session.configured":
      ttsRate = evt.session.output_audio_sample_rate || 48000;
      if (playCtx) playCtx.close();
      playCtx = new AudioContext({ sampleRate: ttsRate });
      playCursor = playCtx.currentTime;
      startMic();
      break;
    case "response.created":
      // Fresh response — drop any leftover scheduled audio (barge-in)
      playCursor = playCtx.currentTime;
      break;
    case "response.output_audio.delta":
      playPCM16(b64ToArrayBuffer(evt.delta));
      break;
    case "error":
      console.error(evt.error);
      break;
  }
};

async function startMic() {
  const stream = await navigator.mediaDevices.getUserMedia({ audio: true });
  micCtx = new AudioContext({ sampleRate: 16000 });
  const src = micCtx.createMediaStreamSource(stream);
  const proc = micCtx.createScriptProcessor(2048, 1, 1); // production: use AudioWorklet
  proc.onaudioprocess = (e) => {
    const float32 = e.inputBuffer.getChannelData(0);
    const pcm16 = floatTo16BitPCM(float32);
    ws.send(JSON.stringify({
      type: "input_audio_buffer.append",
      audio: arrayBufferToBase64(pcm16.buffer),
    }));
  };
  src.connect(proc);
  proc.connect(micCtx.destination);
}

function playPCM16(arrayBuf) {
  const view = new DataView(arrayBuf);
  const samples = arrayBuf.byteLength / 2;
  const buf = playCtx.createBuffer(1, samples, ttsRate);
  const ch = buf.getChannelData(0);
  for (let i = 0; i < samples; i++) ch[i] = view.getInt16(i * 2, true) / 0x8000;
  const src = playCtx.createBufferSource();
  src.buffer = buf;
  src.connect(playCtx.destination);
  const start = Math.max(playCtx.currentTime, playCursor);
  src.start(start);
  playCursor = start + buf.duration;
}

function floatTo16BitPCM(float32) {
  const out = new Int16Array(float32.length);
  for (let i = 0; i < float32.length; i++) {
    const s = Math.max(-1, Math.min(1, float32[i]));
    out[i] = s < 0 ? s * 0x8000 : s * 0x7fff;
  }
  return out;
}

function arrayBufferToBase64(buf) {
  let bin = "";
  const bytes = new Uint8Array(buf);
  for (let i = 0; i < bytes.length; i++) bin += String.fromCharCode(bytes[i]);
  return btoa(bin);
}

function b64ToArrayBuffer(b64) {
  const bin = atob(b64);
  const buf = new Uint8Array(bin.length);
  for (let i = 0; i < bin.length; i++) buf[i] = bin.charCodeAt(i);
  return buf.buffer;
}
```

## Reference implementation

A complete browser client is published at **[github.com/smallest-inc/hydra_agents](https://github.com/smallest-inc/hydra_agents)**.

The reference client demonstrates:

- the full event lifecycle, including barge-in handling and gapless playback,
- a live wire-log of every frame sent and received,
- multi-agent presets (companion, restaurant phone agent, banking concierge),
- client-side tool execution via `function_call_output`,
- editable persona, voice, and instructions at runtime.

```bash
git clone git@github.com:smallest-inc/hydra_agents.git
cd hydra_agents
npm install
npm run dev
# open http://localhost:3000, paste your SMALLEST_API_KEY, click Connect
```

Stack: Next.js 15, React 19, TypeScript, Tailwind CSS. The session state machine lives in `src/app/hooks/useHydraSession.ts`; audio I/O (mic worklet + gapless playback with barge-in) lives in `src/app/lib/audio.ts`.
